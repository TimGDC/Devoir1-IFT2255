Hypothèses

  • Les activités qu'un utilisateur peut participer a sont celles que lui a cree ou celles que d'autres utiliseurs ont crees. Cela permet la participation individuelle et collaborative.
  • Les actions d'une tache sont exécutées dans l'ordre qu'elles apparaissent parmi la liste d'actions

Tâche 3 : Cas d'utilisation

Cas d'utilisation 1 : S'inscrire comme utilisateur.
But : Inscription au logiciel Robotix comme utilisateur qui souhaite gérer des robots/composants.
Préconditions : Avoir nom, prénom, pseudo, adresse courriel, téléphone et nom de la compagnie à disposition.
Acteurs : Utilisateur (principal)
Scénario principal : description étape par étape, ordonnée numériquement
1. L'utilisateur accède à la page d'inscription.
2. L'utilisateur remplit le formulaire d'inscription avec les informations requises (nom, prénom, pseudo, adresse courriel, téléphone, nom de la compagnie).
3. L'utilisateur soumet le formulaire d'inscription.
4. Le système vérifie les informations fournies.
5. Le système enregistre les informations de l'utilisateur dans la base de données.
6. Le système envoie un email de confirmation à l'utilisateur.
7. L'utilisateur clique sur le lien de confirmation dans l'email pour activer son compte.
8. Le système active le compte de l'utilisateur et confirme l'inscription.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
1a.1. Si l'utilisateur ne fournit pas toutes les informations requises, le système affiche un message d'erreur et demande à l'utilisateur de compléter les informations manquantes.
2a.1. Si l'utilisateur fournit une adresse courriel déjà utilisée, le système affiche un message d'erreur et demande à l'utilisateur de fournir une autre adresse courriel.
4a.1. Si le système détecte des informations invalides (ex. format d'email incorrect), il affiche un message d'erreur et demande à l'utilisateur de corriger les informations.
Postconditions : Inscription (compte créé) à Robotix.
Cas d'utilisation 2 : S'inscrire comme fournisseur.
But : Permettre à un nouveau fournisseur de créer un compte sur le système Robotix.
Préconditions : Le fournisseur doit avoir une compagnie enregistrée. Le fournisseur doit avoir accès à une connexion internet.
Acteurs : Fournisseur (principal)
Scénario principal : description étape par étape, ordonnée numériquement
1. Le fournisseur accède à la page d'inscription.
2. Le fournisseur remplit les champs requis : nom (unique), adresse, email, téléphone.
3. Le fournisseur soumet le formulaire.
4. Le système vérifie l'unicité du nom du fournisseur et enregistre les informations.
5. Le système envoie un courriel de confirmation au fournisseur.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
1a. Si le nom du fournisseur n'est pas unique, afficher un message d'erreur et demander un nouveau nom.
Postconditions : Le fournisseur est enregistré et reçoit un courriel de confirmation.
Cas d'utilisation 3 : Enregistrer un robot.
But : Permettre à un utilisateur d'ajouter un nouveau robot à sa flotte dans le système Robotix.
Préconditions : L'utilisateur doit être authentifié dans le système. Le robot doit avoir un numéro de série unique fourni par le fournisseur.
Acteurs : Utilisateur (principal)
Scénario principal : description étape par étape, ordonnée numériquement
1. L'utilisateur accède à la section "Enregistrer un robot".
2. L'utilisateur entre le numéro de série du robot.
3. L'utilisateur attribue un nom et un type au robot.
4. L'utilisateur soumet le formulaire.
5. Le système vérifie l'unicité du numéro de série et enregistre les informations du robot.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
2a. Si le numéro de série n'est pas unique, afficher un message d'erreur et demander un nouveau numéro de série.
Postconditions : Le robot est enregistré et peut être géré par l'utilisateur.
Cas d'utilisation 4 : Acheter une composante.
But : Permettre à un utilisateur d'acheter des composantes pour construire ou améliorer un robot.
Préconditions : L'utilisateur doit être authentifié dans le système.
Acteurs : Utilisateur (principal)
Scénario principal : description étape par étape, ordonnée numériquement
1. L'utilisateur accède à la section "Acheter des composantes".
2. L'utilisateur sélectionne une composante parmi celles disponibles.
3. L'utilisateur lit la description et le prix de la composante.
4. L'utilisateur ajoute la composante à son panier.
5. L'utilisateur vérifie le contenu de son panier et procède au paiement.
6. Le système enregistre l'achat et met à jour l'inventaire de l'utilisateur.
Scénarios alternatifs : ramifications du déroulement principal, alternatives possibles à certaines étapes
2a. Si la composante n'est pas disponible, afficher un message d'erreur.
Postconditions : La composante est ajoutée à l'inventaire de l'utilisateur.
Cas d'utilisation 5 : Gérer les activités maintenues
But : Voir et gérer l'ensemble des activités de ces robots. Il peut voir les details des activités, les annuler, ou les modifier. Permet une gestion facile des activités pour l'utilsateur.
Préconditions : L'utilisateur doit etre inscrit dans Robotix et avoir des activités déja enregistrées
Acteurs : Utilisateur (principal)
Scénario principal :
1. L'utilisateur se connecte a Robotix
2. L'utilisateur navigue vers la section des activités
3. Le systeme affiche sa liste d'activités et les robots associés a chacune
4. L'utilisateur sélectionne une activité a visualiser les détails
5. Le système affiche les details de l'activité
Scénarios alternatifs :
3a.1. Si l'utilisateur n'a pas d'activités liées au robots, le système affiche qu'aucune activité a été trouvée.
4a.1. Il choisit d'annuler une activité
4b.1. Il choisit de modifier une activité
 4b.1.1. Il change les dates de début/fin d'une activité.
Postconditions : L'utilisateur a consulté et eventuellement changé les activités qu'il maintient pour ses robots.
Cas d'utilisation 6 : Participer a une activité
But : S'inscrire et participer à une activité. Permet a l'utilisateur de configurer l'activité a son gout. Notamment choisir une partcipation individuelle ou collaborative (en communauté).
Préconditions : L'utilisateur doit etre connecté au système et avoir une activité créée.
Acteurs : Utilsateur (principal)
Scénario principal :
1. L'utilisateur se connecte au système.
2. L'utilisateur navigue vers les activités.
3. L'utilisateur recherche une activité à laquelle il souhaite participer.
4. Le système affiche les activités disponibles.
5. L'utilisateur sélectionne l'activité et choisit d'y participer.
6. Le système valide la participation de l'utilisateur.
Scénarios alternatifs :
5a.1. Si l'activité est collaborative et que l'activité est pleine (nombre de collaborateurs maximum atteint), système affiche message d'erreur de participation.
5a.2. Le systeme affiche que la participation a l'activité a echouée.
Postconditions :L'utilisateur participe a une activité avec son/ses robots et éventuellement d'autres utilisateurs.
Cas d'utilisation 7 : Allouer une tâche
But : Créer et allouer une tâche à un robot directement ou a travers une activité. Cela depend des capacités du robot. Une association de plusieurs taches specifiques en serie definit une activité.
Préconditions : L'utilisateur doit etre connecté et avoir des robots avec actions ou activités disponibles.
Acteurs : Utilisateur (principal)
Scénario principal :
1. L'utilisateur se connecte au système
2. L'utilisateur navigue vers la gestion des taches des robots
3. Le système affiche les taches disponibles
4. L'utilisateur sélectionne la tache et le robot ou l'activité pour allouer la tâche
 4.1. L'utilisateur selectionne l'option de planification pour cette tache (aucune, horaire specifique, ou action declencheur)
 4a.3. Le système confirme les details de la planification et valide.
5. Le système valide la selection et alloue la tache au robot
Scénarios alternatifs :
4a.1. L'utilisateur creer une nouvelle tache (n'en selectionne pas une existente) avec une sequence d'actions.
4a.2. Le système confirme la tache et l'ajoute a la liste des taches disponibles.
4b.1. Si aucun robot ou activité est disponible, le système affiche que l'allouage n'est pas possible, seule la creation de taches est possible.
Postconditions : Le robot de l'utilisateur est en mesure de performer une tache qui lui a ete allouee
Cas d'utilisation 8 : Creer une Action
But : Créer une action qu'un robot peut exécuter, ceci peut etre un mouvement, son, affichage, etc... Les actions sont indepedantes des robots et peuvent etre assignes aux taches.
Préconditions : Aucune (a part que l'utilisateur soit connecté) car les actions sont indepedantes des robots
Acteurs : Utilisateur (principal)
Scénario principal :
1. L'utilisateur se connecte au système
2. L'utilisateur se rend a la creation d'actions
3. L'utilisateur choisit de creer une nouvelle action
 3.1. Il choisit le type d'action (mouvement, son, voix, ecoute, affichage)
 3.2. Il specifie la nature de l'action.
4. Le système valide l'action et la rajoute a la liste d'action disponibles
Scénarios alternatifs :
4a.1. Le système rejette une action qui n'est pas valide (n'a pas de sens, ou n'est pas correctement specifiee)
Postconditions : L'action est creee, se trouve dans la liste d'actions, et un robot avec les composantes necessaires peut l'effectuée.

Préconditions : Etre connecté au systeme, avoir au moins un robot enregistré.
Acteurs : Utilisateur (principal)
Scénario principal :
1. Le système détecte une erreur avec un robot et envoie une alerte à l'utilisateur.
2. L'utilisateur reçoit la notification via Robotix.
3. L'utilisateur se connecte au système et lit l'alerte d'erreur.
4. Le système affiche les détails de l'anomalie et propose des actions recommandées.
5. L'utilisateur sélectionne une action recommandée pour intervenir.
6. Le système exécute l'action sélectionnée pour tenter de résoudre le problème
7. Le système vérifie si le problème est résolu et notifie l'utilisateur du résultat.
Scénarios alternatifs :
4a.1. L'utilisateur choisit de ne rien faire.
4a.2. L'erreur persiste et le système halte toute activité du robot associé.
7a.1. L'erreur n'est pas resolue, le système transmet le probleme automatiquement a l'equipe de support.
Postconditions : Le problème est résolu ou est en cours de résolution par l'équipe de support. L'utilisateur est informé de l'état final de l'intervention

Tâche 4 : Risques

Nous identifions 5 risques, du plus sevère au moins sevère, qui pourraient donner lieu à une mauvaise éxperience du logiciel de la part de l'utilisateur.

  1. (Risque sévère) Faille dans la sécurité des données : Les utilisateurs se confient à nous avec des informations sensibles (nom, nom de compagnie, adresse courriel, téléphone, informations sur leur(s) robot(s)) lors de l'inscription à Robotix. Si nous ne gérons pas bien ces données, il pourrait y avoir fuite/perte des données. Une mauvaise gestion des données pourrait être du au manque d'encryptage, de backups, ou si nous ne mettons pas assez de restrictions d'accès par exemple. Cela mettrait les utilisateurs à risque, possiblement en danger, et aurait des implications légales importantes.
  2. (Risque important) Fiabilité des fournisseurs et qualité des composantes : Nous nous fions aux fournisseurs pour qu'ils manufacturent des composantes de robots fonctionnels et de qualité (au niveau de la norme industrielle). Cela est critique, car si nous obtenons des composantes médiocres ou même défecteuses, les robots assemblés risqueront de ne pas fonctionner proprement, tomberont en panne plus souvent, et le cout de maintenance sera beaucoup plus élevé.
  3. (Risque important) Efficacité des capteurs : Les capteurs jouent un rôle crucial pour nos robots. Avec les données receuillies (périodiquement) par les capteurs, nous pouvons déterminer toutes sortes d'informations importantes sur l'état du robot. Ceci permet notamment de controler les mouvements du robot, d'évaluer le robot pour des problemes potentiels, et de gérer les differentes activités du robot. Des capteurs inefficaces receuilleront moins de données (par unité de temps), ce qui peut entraîner des erreurs de performance, des déplacements imprécis, et en general un mauvais fonctionnement du robot.
  4. (Risque important) Limitations techniques des robots concernant les Activités : Avec le nombre limité de composantes disponibles, les robots risqueront d'avoir des capacités techniques limitées. Ainsi, certaines activités (Jeux/Education/Creation) que les utilisateurs voudraient ne pourront pas être effectuées par les robots. Cela pourrait décevoir les utilisateurs, réduisant l'engagement et la satisfaction des utilisateurs.
  5. (Risque éloigné) Compatibilité : Étant donné que differents fournisseurs produisent différents composants, il faut s'assurer qu'il y ait compatibilité entre composants, ou clairement indiquer les incompatibilités dans l'inventaire. Une incompatibilité peut entrainer des dysfonctionnements dans les robots, des défaillances dans l'exécution des tâches, et compliquer la maintenance des robots.

Tâche 5 : Exigences non-fonctionnelles

Nous identifions 5 besoins non-fonctionnels du système.

  1. Utilisabilité : Robotix doit avoir une interface organisée pour favoriser son utilisation. Etant donné qu'il y beaucoup de différentes fonctionnalités, faciliter leur acces et manipulation est important pour l'utilisabilité du logiciel. Notamment, l'enregistrement de robots, le contrôle du mouvement des robots, la création d'activités/de taches, l'affichage de l'état des robots, et la consultation des composantes devraient être des parties clairement separées. L'interface doit permettre a quiconque de naviguer le logiciel facilement.
  2. Fiabilité : Le système doit être capable de fonctionner de manière continue sans interruptions. Les utilisateurs doivent pouvoir accéder aux fonctionnalités a tout temps, et les requêtes faites doivent pouvoir être traitées sans soucis. La fiabilité est importante pour assurer une certaine fluidité des operations et aider a garder la confiance des utilisateurs.
  3. Maintenabilité : Le logiciel doit être facilement maintenable (resoudre bugs, mises a jour, revisions, ...). Ceci peut être anticiper lors de la conception, lorsqu'il s'agit de définir les modules du logiciel. L'utilisation de modules permet de compartimentaliser le logiciel en petites parties, chacune bien encapsulée. Puis, lors de la maintenance, il suffit de trouver le module a réviser et retravailler seulement ce module, laissant le reste.
    Pour un logiciel avec autant de fonctionnalités, et étant donné qu'il s'agit de robots (technologies complexes avec beaucoup de composants), les bugs sont inévitables, donc il est important d'avoir une bonne maintenabilité.
  4. Extensibilité (Scalability) : Robotix doit pouvoir soutenir un grand nombre d'utilisateurs sur sa platforme, sans de perte de performance. De plus, chaque utilisateur peut avoir plusieurs robots, ce qui signifie qu'il y aura beaucoup plus de robots a gérer que d'utilisateurs. Les algorithmes, structures de données, et bases de données utilisées doivent donc être capables de traiter une augmentation (possiblement soudaine) d'utilisateurs et robots inscrits sur Robotix.
  5. Performance : Le système doit avoir un temps de réponse ainsi qu'une éxecution rapide. Pour chaque requête faite par l'utilisateur (client ou fournisseur), il faut que le logiciel y réponde l'éxecute rapidement. Le délai entre une action de l'utilisateur et sa réalisation doit être minimale. Notamment pour le controle des mouvements, un trop long délai entre l'appui d'un bouton et le mouvement du robot peut être vu comme désagreable en tant qu'utilisateur.
    Ceci peut être assurer avec un bon choix d'algorithmes et structures de données.
